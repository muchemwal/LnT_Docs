# Compliance Closure

import { Callout } from 'nextra/components'

After handover, the PTF transitions to `compliance_pending`. The crew completes a closure form.

## Completion Checklist

| Field | Type | Description |
|---|---|---|
| Vehicle cleaned | Boolean | Infection control confirmation |
| Incident reported | Boolean | Links to incident reporting if yes |
| Delay reason | Free text | If applicable |
| Route used | Free text | Description of route taken |
| Declaration signed | Boolean | Crew confirms all checks complete |

![Compliance Checklist](/images/46-compliance-checklist.png)
![Compliance Declaration](/images/47-compliance-declaration.png)

## Final Closure

Once submitted, the PTF transitions to `closed` — a terminal state. The `job_closed_at` timestamp
is set.

Closure triggers post-job automation:
- Automatic invoice generation (if rate card exists for the client)
- Timesheet period update with worked hours
- **Factorial HR shift sync** (clock_in / clock_out pushed to Factorial) or **Deel time tracking sync**
- Audit trail finalisation
- `NotificationReceipt` creation for delivery confirmation

![PTF Closed](/images/48-ptf-closed-status.png)

## Compliance Service (Docker-Only)

The compliance-service (port 3016) runs exclusively in Docker and handles automated compliance tasks:

| Feature | Description |
|---|---|
| Breach detection | Automated scanning for data breach indicators |
| Retention enforcement | Policy-based data lifecycle management |
| Retention policy | 8-year clinical record minimum retention |
| Data processor registry | Third-party processor tracking |

<Callout type="info">
  The compliance-service is not included in local development health checks. It communicates
  with audit-service (3015) and notification-service (3011) for alerting.
</Callout>

## Incident Reporting

If an incident occurred during transfer:

| Field | Description |
|---|---|
| `incident_type` | `patient_fall`, `near_miss`, `vehicle_issue`, `clinical_concern`, `other` |
| `severity` | `standard` or `serious_adverse_event` |
| Location / description | Encrypted |
| Persons involved | Encrypted JSON |
| Investigation | Assigned investigator, tracked through `open → under_review → resolved` |
