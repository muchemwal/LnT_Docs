# Booking Lifecycle (PTF)

import { Callout, Steps, Tabs, Tab } from 'nextra/components'

Bookings are created through a **7-step Patient Transfer Form (PTF) wizard**. The form creates both
a `TransferJob` record and an associated `PTF` record.

## PTF Multi-Step Form

![PTF List Page](/images/06-ptf-list-page.png)

<Steps>

### Step 1 — Referral

Referral source, client/funding authority dropdown, booking date (defaults to today).

- Internal users select from a client dropdown which auto-fills the funding authority
- External users have referral source locked to "External Client" with funding authority prepopulated

![Step 1 Referral](/images/08-step1-referral.png)

### Step 2 — Patient

Patient name, DOB (calendar picker), NHS number with auto-lookup, gender, ethnicity, blood type,
allergies, medications, GP details, next of kin.

<Callout type="info">
  When a previously used NHS number is entered, the system auto-populates name, DOB, blood type,
  GP details, and next of kin from the most recent record.
</Callout>

![Step 2 Patient](/images/10-step2-patient-empty.png)
![NHS Lookup](/images/11-step2-nhs-lookup.png)

### Step 3 — Transfer

Pickup and destination addresses with postcodes. Mileage is auto-calculated via geocoding.

- Return leg checkbox reverses addresses and doubles mileage
- Transfer datetime, priority (routine/urgent/emergency), mode of arrival
- **Multi-leg journeys**: add transport legs with segment types (pickup, travel, wait_at_facility, appointment, fuel_stop, meal_stop, rest_stop, accommodation, shift_change, return_transport, destination_handover)

![Mileage Calculated](/images/15-step3-mileage-calculated.png)
![Return Leg](/images/16-step3-return-leg.png)

### Step 4 — Clinical

Clinical notes (encrypted), mobility requirements, equipment, infection control, oxygen, MHA section.

All clinical notes are stored with **AES-256 encryption** at rest.

![Clinical Step](/images/18-step4-clinical.png)

### Step 5 — Risk Assessment

Risk checklist with toggle switches: violence, self-harm, absconding, substance misuse.
Each risk can have accompanying notes.

**Safeguarding concerns** are recorded separately via `PTFSafeguarding` when applicable.

![Risk Assessment](/images/19-step5-risk-checklist.png)

### Step 6 — Crew & Escort

- Internal users: crew dropdown, vehicle dropdown, escort options
- External users: free-form crew name textarea
- **Assign Team** dropdown — selecting a crew team auto-fills all member slots
- Number of crew (1-4) with individual crew member selects and lead designation
- **Multi-crew assignment**: roles (driver, medic, escort, supervisor, relief_driver)
- Vehicle assignment
- Escort required (with escort count and names)

![Crew Internal](/images/20-step6-crew-internal.png)

### Step 7 — Review & Submit

Full booking summary. Submit creates the PTF in `draft` status.

![Review](/images/22-step7-review.png)

</Steps>

## Status Transitions

```
draft → booking → crew_assigned → enroute_pickup → arrived_pickup
→ departed_pickup → enroute_destination → arrived_destination
→ handover_completed → compliance_pending → closed
```

Each transition:
- Validates against `VALID_TRANSITIONS` (forward-only)
- Sets a corresponding timestamp (e.g. `crew_assigned_at`, `pickup_arrival_at`)
- Creates a `WorkflowEvent` audit record
- Triggers push notifications (job reference only, never patient data)

## Job Status (Transfer Job)

The `TransferJob` has its own status machine:

```
pending → accepted → en_route → on_scene → at_pickup → patient_loaded
→ in_transit → at_destination → completed (or cancelled)
```

| Status | Description |
|---|---|
| `pending` | Job created, awaiting crew assignment |
| `accepted` | Crew accepted the job |
| `en_route` | Crew en route to pickup |
| `on_scene` | Crew arrived in area |
| `at_pickup` | Crew at pickup location |
| `patient_loaded` | Patient loaded into vehicle |
| `in_transit` | Active transit to destination |
| `at_destination` | Arrived at destination |
| `completed` | Job successfully completed |
| `cancelled` | Job cancelled |

## Multi-Leg Journey Support

For long-distance or complex transfers, jobs can have multiple transport legs and segments:

| Model | Purpose |
|---|---|
| `JobTransportLeg` | A single leg of a multi-leg journey (e.g. Hospital A → Hotel → Hospital B) |
| `JobSegment` | Granular segment within a leg (pickup, travel, wait, appointment, etc.) |
| `JobAccommodation` | Accommodation records for overnight stays |
| `JobMeal` | Meal records during long journeys |
| `JobExpense` | Expense tracking (fuel, parking, tolls, etc.) |

### Segment Types

| Type | Description |
|---|---|
| `pickup` | Collecting patient at origin |
| `travel` | Active driving segment |
| `wait_at_facility` | Waiting at a medical facility |
| `appointment` | Patient attending an appointment |
| `fuel_stop` | Refuelling stop |
| `meal_stop` | Meal break |
| `rest_stop` | WTR-mandated rest break |
| `accommodation` | Overnight stay |
| `shift_change` | Crew changeover point |
| `return_transport` | Return journey |
| `destination_handover` | Handover at final destination |

## Multi-Crew Assignments

Complex transfers support multiple crew members via `JobCrewAssignment`:

| Field | Description |
|---|---|
| `crew_role` | `driver`, `medic`, `escort`, `supervisor`, `relief_driver` |
| `duty_status` | `active_transport`, `standby_waiting`, `off_duty_rest`, `accommodation_rest` |
| `is_lead` | Boolean — primary responsible crew member |

## Return Journey Logic

| Scenario | Behaviour |
|---|---|
| Return leg disabled | Single outbound job + PTF |
| Return leg enabled | Two jobs created: A→B and B→A, each with own PTF |
| Mileage | Doubled when return leg is enabled |
| Dispatch | Both jobs appear independently on the dispatch board |
| Billing | Each job generates a separate invoice |

## PHI Encryption

All Patient Health Information is encrypted with AES-256 before database write via `encryptPhiFields()`:

```
patient_name, patient_dob, patient_nhs_number, patient_blood_type,
patient_allergies, patient_medications, patient_diagnosis,
gp_name, gp_surgery, next_of_kin_name, next_of_kin_phone,
clinical_notes, handover_notes, transfer_origin, transfer_destination
```

Decryption happens only at the API response layer via `decryptPtfRecord()`.

## Medication Administration

### Recording Medications (PTF Detail page)

The PTF detail page includes a **Medication Administration Log** section:
- Table: medication name, dose, route, given at, next due, notes
- Overdue medications highlighted in red with ⚠️ indicator
- "💊 Record Administration" button opens inline form

### Recording Form Fields

| Field | Description |
|---|---|
| Medication name | Required |
| Dose | e.g. "1g", "500mg" |
| Route | oral, IV, IM, SC, topical, inhaled, rectal, sublingual, other |
| Given at | Datetime (required) |
| Next due at | Datetime (optional — triggers crew notifications) |
| Notes | Optional free text |

### Mobile Medication Recording

During active jobs, crew can record medication via the job detail screen:
- Select from patient's known medications (from PTF)
- Record: medication name, dose, route, time given, next due
- Push notification 5 minutes before `next_due_at`

### Due Medications Query

`GET /v1/medication-administrations/due?minutes=5` returns all medications due within the specified window, used by the notification system.


## MHA Compliance Fields

For Mental Health Act transfers, additional compliance fields are captured:
- Authority to convey
- Medical recommendations
- AMHP (Approved Mental Health Professional) application
- Section documentation references
