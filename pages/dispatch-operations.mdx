## Dispatch Board

The dispatch board provides a real-time pipeline view with columns: **pending → accepted → en_route
→ on_scene → at_pickup → patient_loaded → in_transit → at_destination → completed**. Updates arrive via WebSocket within seconds of crew actions.

![Dispatch Board](/images/25-dispatch-board-pipeline.png)

## Kong API Gateway

All dispatch traffic routes through **Kong 3.7** (DB-less declarative config):

| Feature | Configuration |
|---|---|
| Rate limiting | 100 requests/minute per consumer |
| CORS | Configured for web + mobile origins |
| IP restriction | Admin/audit routes restricted |
| Request size | Configurable limits per route |
| SSL | TLS termination on port 8443 |

## Crew Assignment

Assignment is performed through a modal with:
- Crew member dropdown (filtered to active, compliant crew)
- Vehicle dropdown (filtered to compliant vehicles)
- Compliance check indicators (DBS status, certifications, MOT/insurance)
- **Multi-crew support**: assign multiple crew with roles (driver, medic, escort, supervisor, relief_driver)

![Crew Assignment](/images/27-crew-assignment-modal.png)

<Callout type="warning">
  If crew DBS or vehicle MOT/insurance is expired, the assignment is rejected with a
  `COMPLIANCE_VIOLATION` error.
</Callout>

On successful assignment, the crew receives a push notification within 5 seconds (via APNs/FCM).
An unaccepted job triggers a dispatcher alert after 10 minutes via Redis tracking.

## GPS Architecture

```
Mobile → WSS → Caddy → GPS Service (port 3004)
                            ├── Persist to Postgres (gps_locations table)
                            ├── Broadcast to Dispatch Board (WebSocket)
                            └── Smart auto-transition (geofence triggers)
```

### GPS Auto-Transition

When GPS indicates the crew has arrived at a location:
- Within geofence radius of pickup → auto-suggests `at_pickup` status
- Within geofence radius of destination → auto-suggests `at_destination` status
- Crew confirms or dismisses the prompt

## Notifications

| Channel | Mechanism | PHI Policy |
|---|---|---|
| Push (iOS) | APNs | Job reference only, never patient data |
| Push (Android) | FCM | Job reference only, never patient data |
| SMS | Twilio | Job reference only |
| In-app | SSE stream at `GET /v1/notifications/stream` | Scoped to user role |
| Email | SMTP | Non-PHI summaries only |

### Notification Categories

`job_alert`, `rota_change`, `cert_expiry`, `system_alert`, `message`, `incident_alert`, `billing`

### Notification Receipts

Each notification has a `NotificationReceipt` tracking delivery status across channels.

### Key Automated Notifications

- Crew assigned → crew notified (push + in-app)
- Crew en route / arrived / departed → dispatcher notified
- Handover completed → both parties notified
- Unaccepted job after 10 min → dispatcher alerted
- Certification expiring → employee + HR notified
- Invoice overdue → administrator alerted

## Status Machine

Valid transitions (defined in `status-machine.ts`):

| From | To (allowed) |
|---|---|
| pending | accepted, cancelled |
| accepted | en_route, cancelled |
| en_route | at_pickup, cancelled |
| at_pickup | in_transit, cancelled |
| in_transit | at_destination, completed, cancelled |
| at_destination | completed, cancelled |
| completed | (terminal) |
| cancelled | (terminal) |

Legacy aliases: `on_scene` = `at_pickup`, `patient_loaded` = `in_transit`

## Geocode Endpoint

`POST /v1/geocode` — Geocode an address to lat/lng coordinates. Used for pickup/destination geocoding during job creation.
