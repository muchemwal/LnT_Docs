# Factorial HR & Deel Integration

import { Callout, Tabs, Tab, Steps } from 'nextra/components'

The platform integrates with [Factorial HR](https://apidoc.factorialhr.com/reference) via the
**2026-01-01 API version** and with [Deel](https://developer.deel.com/) via the **REST v2 API**
for employee management, attendance tracking, shift synchronisation, absence management, and
timesheet approvals.

<Callout type="info">
  Service: `hr-integration` (port 3017). Connectors: `FactorialConnector` and `DeelConnector`,
  both implementing the `HRPort` interface from `@landtts/shared`.
</Callout>

## Architecture

The hr-integration service uses a **port/adapter pattern** allowing interchangeable HR providers:

```
┌─────────────────────────────────────────────────────┐
│  hr-integration service (port 3017)                  │
│                                                     │
│  ┌─────────────┐    ┌──────────────────────────┐   │
│  │   HRPort    │◄───│  FactorialConnector      │   │
│  │  interface  │    │  (2026-01-01 API)        │   │
│  │             │    └──────────────────────────┘   │
│  │             │                                   │
│  │             │    ┌──────────────────────────┐   │
│  │             │◄───│  DeelConnector           │   │
│  │             │    │  (REST v2 API)           │   │
│  └─────────────┘    └──────────────────────────┘   │
└─────────────────────────────────────────────────────┘
```

<Steps>

### Create OAuth Application

Navigate to `https://api.demo.factorial.dev/oauth/applications` and create credentials.

### Authorise

```
https://api.demo.factorial.dev/oauth/authorize?client_id=CLIENT_ID&redirect_uri=urn:ietf:wg:oauth:2.0:oob&response_type=code
```

### Exchange for Token

```bash
curl -X POST 'https://api.factorialhr.com/oauth/token' \
  -d 'client_id=<CLIENT_ID>&client_secret=<CLIENT_SECRET>&code=<AUTH_CODE>&grant_type=authorization_code&redirect_uri=<REDIRECT_URI>'
```

Response:
```json

```


## Factorial — Attendance Shifts API

The core integration point for crew timesheets. Maps directly to the mobile app's shift check-in/out.

### Read All Shifts

```
GET /api/2026-01-01/resources/attendance/shifts
```

Query parameters:
| Parameter | Type | Description |
|---|---|---|
| `employee_ids[]` | `integer[]` | Filter by employee IDs |
| `start_on` | `string` | Shifts starting after this date |
| `end_on` | `string` | Shifts ending before this date |
| `half_day` | `boolean` | Filter half-day shifts |
| `workable` | `boolean` | Filter workable day shifts |
| `latest_shift` | `boolean` | Only latest shift per employee |

Response shape:
```json
{
  "data": [
    {
      "id": 123,
      "employee_id": 456,
      "date": "2026-03-17",
      "reference_date": "2026-03-17",
      "clock_in": "08:00",
      "clock_out": "17:00",
      "observations": "Patient transfer shift",
      "location_type": "business_trip",
      "minutes": 540,
      "workable": true,
      "company_id": 1
    }
  ],
  "meta": {
    "has_next_page": false,
    "total": 1
  }
}
```

### Create a Shift (Push Timesheet)

```
POST /api/2026-01-01/resources/attendance/shifts
```

### Clock In / Clock Out

```
POST /api/2026-01-01/resources/attendance/shifts/clock_in
POST /api/2026-01-01/resources/attendance/shifts/clock_out
```

### Break Management

```
POST /api/2026-01-01/resources/attendance/shifts/break_start
POST /api/2026-01-01/resources/attendance/shifts/break_end
```

## Mobile App Actions → HR Provider Mapping

| Mobile Action | Platform Event | Factorial API Call | Deel API Call |
|---|---|---|---|
| Crew taps "Check In" | Shift record created | `POST /attendance/shifts/clock_in` | `POST /rest/v2/time-tracking` |
| Crew taps "Start Break" | Break start logged | `POST /attendance/shifts/break_start` | N/A (included in shift) |
| Crew taps "End Break" | Break end logged | `POST /attendance/shifts/break_end` | N/A |
| Crew taps "Check Out" | Shift completed | `POST /attendance/shifts/clock_out` | `PATCH /rest/v2/time-tracking/:id` |
| PTF reaches `closed` | Auto-timesheet generated | `POST /attendance/shifts` | `POST /rest/v2/time-tracking` |
| HR approves timesheet | Timesheet approved | `POST /attendance/reviews/bulk_create` | `POST /rest/v2/payroll` |
| Crew requests absence | Absence record created | `POST /timeoff/leaves` | `POST /rest/v2/time-off` |

```typescript
// POST /api/2026-01-01/resources/attendance/shifts
async syncShifts(shifts: HRShift[]): Promise<SyncResult> {
  return withRetry(async () => {
    for (const shift of shifts) {
      await axios.post(
        `${FACTORIAL_API_BASE}/resources/attendance/shifts`,
        {
          employee_id: shift.factorial_employee_id,
          clock_in: shift.start_at,
          clock_out: shift.end_at,
          reference_date: shift.reference_date,
          observations: shift.role ?? '',
        },
        { headers: this.headers },
      );
    }
    return { success: true, syncedAt: new Date().toISOString() };
  });
}
```

```typescript
// POST /rest/v2/workers
async syncEmployee(employee: HREmployee): Promise<SyncResult> {
  return withRetry(async () => {
    const payload = {
      first_name: employee.full_name.split(' ')[0] ?? '',
      last_name: employee.full_name.split(' ').slice(1).join(' ') ?? '',
      job_title: employee.role,
      start_date: employee.employment_start_date,
      weekly_hours: employee.contracted_hours_per_week,
      external_id: employee.id,
    };
    const response = await axios.post(
      `${DEEL_API_BASE}/workers`,
      payload,
      { headers: this.headers },
    );
    return {
      success: true,
      externalId: response.data.data.id,
      syncedAt: new Date().toISOString(),
    };
  });
}
```

## Webhook Support (Factorial)

Factorial supports webhooks for real-time event notifications. Relevant events:

| Webhook | Use Case |
|---|---|
| `Attendance > Shift > Clock ins` | Confirm clock-in synced |
| `Attendance > Shift > Clock outs` | Confirm clock-out synced |
| `Attendance > Shift > Creates` | New shift created externally |
| `Attendance > Shift > Updates` | Shift modified in Factorial |
| `Timeoff > Leave > Approves` | Leave approved in Factorial → update platform |
| `Timeoff > Leave > Rejects` | Leave rejected → notify crew |
| `Employees > Employee > Updates` | Employee data changed → sync back |

Configure via `POST /api/2026-01-01/resources/api-public/webhook-subscriptions`.

## Retry Logic

Both connectors implement `withRetry` from `@landtts/shared`:
- Up to 3 retries with exponential backoff (1s, 2s, 4s)
- On final failure: emits HR Manager notification via notification-service
